Sharding

Introduction to Computer Systems | C++ | Spring 2020

Project Overview

I worked on sharding as a project individually for two weeks. The task of this project was to build a distributed, sharded key-value storage system, using gRPCs and protobufs. In other words, the data stored would be split up across multiple servers, via "shards" (or a range of keys in a key-value store).

Step 1: Creating a simple server

The first step to the project was creating a simple server that could support the Remote Procedure Calls (RPCs) Get, Put, and Append. I decided to use a map to store the server's data as well as a mutex to protect the data during concurrent accesses to the server.



Step 2: Creating a controller program

The second step was creating a shardmaster (or a controller) that would assign shards to the active servers. The RPCs include Join, Leave, Move (moves shards to a different server), and Query (returns the current distribution). The difficult part about this step was figuring out how to evenly distribute the shards when servers joined or left. For example, if the key space was 0-100, and there were three servers — how would the shards be divided evenly?

Writing the RPC Move was tricky because the code had to move the shards and extracting the correct shards without interfering with the remaining shards. A bug I ran into was when the code would be iterating through a server's shards, the code was also simultaneously trying to remove the shard; however once the shard was removed, the number of iterations to take would change — and it would not reach the last shard. To solve this issue, I created a separate data structure to store the new shards and then I copied over the old shards as well. After completing the shardmaster, it would interact with the previous simple server from step 1.

Step 3: Creating a more advanced server

The last step of this project was making an improved server, one that actually moves the shards (and its associated key-value pairs) to the correct server (according to the shardmaster). The function mainly responsible for this task was Query, in which the current server would query the shardmaster and make sure that the shards on the current server are the appropriate shards, and if the server is not responsible for a certain key, then it will "put" the key to the correct server. An interesting bug that I ran into when I was writing Query was when you found a key that a server wasn't responsible for, instead of trying to find the correct server, I ended up just "putting" the shard to the current server. After completing this step of the process, the shardmaster and servers were able to respond to requests to move shards — and the data would be moved as well.
