WeensyOS

Introduction to Computer Systems | C++ | Spring 2020

Project Overview

WeensyOS, a project from the class CS131, was centered around creating a very basic operating system, by implementing the basic virtual and physical memory architecture as well as the system calls fork and exit. We used QEMU to visualize how our code looked like.

The most difficult part of this project was being unable to use GDB to debug the project. When the code didn't work, we would just receive panics or page faults, which were extremely unhelpful in trying to figure out what exactly wasn't working. What I found helpful in debugging was talking over my code with someone else and try to articulate what my code was trying to do and making sure it was fulfilling those tasks (because many times it just felt like I was writing code into a void —— nothing was working it the way I wanted it to).

The First Two Steps

The first step of the project was making sure that no process could write to the kernel or allocate pages of memory from the kernel. This step was just changing how the kernel was initialized and setting the permission flags (is the page writable? user accessible? and present?) correctly. The second part of this step was ensuring that the pages the user requested were not in the region of kernel memory.

The second step of the project was making sure that each process could not write to the other processes running on the operating system.

The second step was by far one of the hardest steps.

The Third and Fourth Step

The third and fourth step of the project was implementing virtual page allocation, so the physical to virtual memory mappings were no longer 1 to 1 and each process were using the same virtual address space (but backed by different physical addresses).

The Last Steps

The last couple of steps involved writing the system call fork and exit.
